# Zero-Config Instrumentation

Observer OS is designed to require **zero code changes** to your application. This document explains how that works.

---

## The problem with SDK-first instrumentation

Traditional observability tools require you to:

1. Install a package
2. Import it in your app
3. Initialize it with config
4. Add middleware/wrappers to every library
5. Create a session before starting
6. Connect the SDK to the session

That's 6 steps before you see a single event. Worse — every developer on the team has to repeat this setup.

Observer solves this differently.

---

## How zero-config works

### 1. Daemon auto-creates a session

When the daemon starts, it checks for active sessions. If none exist, it creates a `Default Session` automatically.

```bash
pnpm --filter @observer-os/daemon start
# Observer OS ready → http://localhost:4000
# [auto] created Default Session ses_abc123
```

No `POST /api/sessions` needed.

### 2. `observer run` injects instrumentation

```bash
observer run node server.js
observer run npm start
observer run tsx src/index.ts
observer run pnpm dev
```

Internally, `observer run` does:

```bash
NODE_OPTIONS="--require /path/to/@observer-os/auto-instrument/dist/index.js" \
OBSERVER_URL="http://localhost:4000" \
  node server.js
```

The `--require` flag causes Node.js to execute `auto-instrument` **before any user code runs**. At that point, `auto-instrument`:

1. Patches `http.createServer` — intercepts ALL incoming HTTP (any framework)
2. Patches `http.request` / `https.request` — intercepts outgoing HTTP
3. Probes `node_modules` — patches whatever it finds (pg, mysql2, ioredis, ws)
4. Connects to `GET /api/sessions/default` async
5. Queues any events emitted before connection, then flushes them

Your application code is **unchanged**. Express, Fastify, Koa, raw `http` — all instrumented the same way.

### 3. Browser inject via one script tag

```html
<script src="http://localhost:4000/observer.js"></script>
```

The inject script is generated by the daemon at request time. It:

- Detects its own URL from the `host` header — no hardcoded URLs
- Fetches `GET /api/sessions/default` to get the active session ID
- Patches `window.fetch`, `XMLHttpRequest`, `WebSocket`, `console.error`, `window.onerror`, `unhandledrejection`, and `history.pushState`
- Queues events until the session ID is resolved, then flushes

No npm install. No configuration. No session ID to pass manually.

## What gets detected automatically

`auto-instrument` uses safe `require()` probing — if a package isn't installed, the patch is a no-op:

```
node_modules/pg          → patch Pool.prototype.query
node_modules/ioredis     → patch prototype.sendCommand
node_modules/redis       → patch RedisClient.prototype.sendCommand
node_modules/ws          → patch WebSocketServer.prototype.on
```

HTTP server and client are always patched (built into Node.js).

Detection takes ~5ms at startup.

---

## The `/api/sessions/default` endpoint

Both the Node.js auto-instrument and the browser inject script hit this endpoint:

```
GET /api/sessions/default
```

Returns the most recently started `ACTIVE` session. If no active session exists, creates one and returns 201.

This means the inject script and the server-side instrumentation automatically share the same session — correlation works out of the box.

---

## Zero-config vs manual SDK

| | Zero-config (`observer run` + inject script) | Manual SDK |
|--|----------------------------------------------|------------|
| Code changes | None | Add imports + middleware |
| Setup time | ~30 seconds | 10–30 minutes |
| Control | Framework-level patching | Per-library fine-grained |
| PII safety | Basic (URL + status only) | Configurable per plugin |
| Cross-domain correlation | Automatic (W3C traceparent) | Automatic |
| Custom event types | Not supported | Full SDK access |
| CI/CD safe | Yes (`OBSERVER_URL` env var) | Yes |

Use zero-config for development and debugging. Use the manual SDK when you need custom event types, PII filtering, or production-safe instrumentation.

## Multiple processes

If your app has multiple services, run each with `observer run`:

```bash
# Terminal 1
observer run node api-server.js

# Terminal 2
observer run node worker.js

# Terminal 3
observer run node websocket-server.js
```

All three processes connect to the same default session. Events from all three appear in one graph.

## Configuring observer run

```bash
# Custom daemon URL
observer run --url http://my-server:4000 node server.js

# With API key
observer run --key my-secret-key node server.js

# Both
observer run --url http://localhost:4000 --key abc123 node server.js
```

Or via environment:

```bash
export OBSERVER_URL=http://localhost:4000
export OBSERVER_API_KEY=my-secret-key
observer run node server.js
```

---

## Disabling auto-instrument selectively

If you need to run without instrumentation:

```bash
# Normal node (no Observer)
node server.js

# Or clear NODE_OPTIONS if already set
NODE_OPTIONS="" node server.js
```

## How events flow

```
Your Node.js app (via observer run)
  └── auto-instrument (--require, runs first)
        ├── patches http.createServer
        ├── patches http.request
        ├── patches pg.Pool.query
        └── connects to daemon → queues events → flushes on connect
                                        ↓
                              POST /api/sessions/{id}/events
                                        ↓
                              Observer Daemon :4000
                                  EventLog → ProjectionEngine
                                        ↓
                              WebSocket push to Explorer

Your Browser
  └── <script src="http://localhost:4000/observer.js">
        ├── patches window.fetch
        ├── patches XMLHttpRequest
        ├── patches WebSocket
        └── connects to /api/sessions/default → flushes queue
                                        ↓
                              POST /api/sessions/{id}/events
                                        ↓
                              Observer Daemon :4000
```
